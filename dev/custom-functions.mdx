PostgreSQL supports four kinds of user-defined functions: SQL functions, C-language functions, procedural language functions (PL/pgSQL, PL/Python, etc.), and internal functions. This page covers SQL functions and C-language functions.

## SQL functions

SQL functions execute an arbitrary list of SQL statements and return the result of the last query.

```sql
CREATE FUNCTION add_em(x integer, y integer) RETURNS integer AS $$
    SELECT x + y;
$$ LANGUAGE SQL IMMUTABLE STRICT;

SELECT add_em(1, 2);  -- 3
```

The body can reference arguments positionally (`$1`, `$2`) or by name when named in the signature.

### SQL set-returning functions

Declare the return type as `SETOF type` or `RETURNS TABLE(...)` to return multiple rows:

```sql
CREATE FUNCTION getfoo(int)
    RETURNS SETOF foo AS $$
    SELECT * FROM foo WHERE fooid = $1;
$$ LANGUAGE SQL;

SELECT * FROM getfoo(1);
```

With output parameters:

```sql
CREATE FUNCTION sum_and_product(x int, y int,
    OUT s int, OUT p int)
RETURNS SETOF record AS $$
    SELECT $1 + $2, $1 * $2;
$$ LANGUAGE SQL;
```

### SQL variadic functions

Mark the last parameter `VARIADIC` to accept a variable number of arguments of the same type:

```sql
CREATE FUNCTION mleast(VARIADIC arr numeric[]) RETURNS numeric AS $$
    SELECT min($1[i]) FROM generate_subscripts($1, 1) g(i);
$$ LANGUAGE SQL;

SELECT mleast(10, -1, 5, 4.4);  -- -1
SELECT mleast(VARIADIC ARRAY[10, -1, 5, 4.4]);  -- pass an array directly
```

### Function volatility

| Volatility | Meaning |
|------------|---------|
| `VOLATILE` (default) | May return different results for the same arguments; can have side effects |
| `STABLE` | Returns the same result for the same arguments within a single statement; cannot modify the database |
| `IMMUTABLE` | Returns the same result for the same arguments forever; the planner can pre-evaluate calls with constant arguments |

Label your functions with the strictest valid category. Any function with side effects must be `VOLATILE`.

## C-language functions

C functions are compiled into shared libraries and loaded on demand by the server.

### Required includes and magic block

Every shared library must include `PG_MODULE_MAGIC` in exactly one source file:

```c
#include "postgres.h"
#include "fmgr.h"

PG_MODULE_MAGIC;
```

The magic block encodes the major version so the server can reject incompatible libraries at load time.

### Version-1 calling convention

All user-defined C functions must use the version-1 calling convention:

```c
PG_FUNCTION_INFO_V1(my_function);

Datum
my_function(PG_FUNCTION_ARGS)
{
    /* ... */
}
```

`PG_FUNCTION_INFO_V1(name)` declares the function and registers it as version-1. The actual function signature is always `Datum funcname(FunctionCallInfo fcinfo)`, but you always write `PG_FUNCTION_ARGS` as the parameter list — it expands to `FunctionCallInfo fcinfo`.

### PG_GETARG_* macros

Arguments are fetched with type-specific macros. Counting starts at 0.

| Macro | C type | SQL type |
|-------|--------|----------|
| `PG_GETARG_INT32(n)` | `int32` | `integer` |
| `PG_GETARG_INT64(n)` | `int64` | `bigint` |
| `PG_GETARG_FLOAT4(n)` | `float4` | `real` |
| `PG_GETARG_FLOAT8(n)` | `float8` | `double precision` |
| `PG_GETARG_BOOL(n)` | `bool` | `boolean` |
| `PG_GETARG_OID(n)` | `Oid` | `oid` |
| `PG_GETARG_TEXT_PP(n)` | `text *` | `text` |
| `PG_GETARG_BYTEA_PP(n)` | `bytea *` | `bytea` |
| `PG_GETARG_CSTRING(n)` | `char *` | `cstring` |
| `PG_GETARG_POINTER(n)` | `void *` | any pointer type |
| `PG_GETARG_DATUM(n)` | `Datum` | any |

For nullable (non-strict) functions, check `PG_ARGISNULL(n)` before calling `PG_GETARG_*`. Do not call `PG_GETARG_*` on a null argument.

### PG_RETURN_* macros

| Macro | Returns |
|-------|---------|
| `PG_RETURN_INT32(x)` | `integer` |
| `PG_RETURN_INT64(x)` | `bigint` |
| `PG_RETURN_FLOAT4(x)` | `real` |
| `PG_RETURN_FLOAT8(x)` | `double precision` |
| `PG_RETURN_BOOL(x)` | `boolean` |
| `PG_RETURN_OID(x)` | `oid` |
| `PG_RETURN_TEXT_P(x)` | `text` |
| `PG_RETURN_BYTEA_P(x)` | `bytea` |
| `PG_RETURN_CSTRING(x)` | `cstring` |
| `PG_RETURN_POINTER(x)` | pointer type |
| `PG_RETURN_VOID()` | `void` |
| `PG_RETURN_NULL()` | NULL (any type) |

### Strict functions

Declare a function `STRICT` to have PostgreSQL return NULL automatically whenever any input argument is NULL, without calling the function body:

```sql
CREATE FUNCTION add_one(integer) RETURNS integer
    AS '$libdir/myext', 'add_one'
    LANGUAGE C STRICT;
```

Without `STRICT`, use `PG_ARGISNULL(n)` to detect null inputs:

```c
if (PG_ARGISNULL(0))
    PG_RETURN_NULL();
```

### Complete C example

```c
#include "postgres.h"
#include <string.h>
#include "fmgr.h"
#include "varatt.h"

PG_MODULE_MAGIC;

/* By-value type example */
PG_FUNCTION_INFO_V1(add_one);

Datum
add_one(PG_FUNCTION_ARGS)
{
    int32 arg = PG_GETARG_INT32(0);
    PG_RETURN_INT32(arg + 1);
}

/* Variable-length type example */
PG_FUNCTION_INFO_V1(copytext);

Datum
copytext(PG_FUNCTION_ARGS)
{
    text *t = PG_GETARG_TEXT_PP(0);
    /*
     * VARSIZE_ANY_EXHDR is size minus the varlena header.
     * Allocate the copy with a full-length header.
     */
    text *new_t = (text *) palloc(VARSIZE_ANY_EXHDR(t) + VARHDRSZ);
    SET_VARSIZE(new_t, VARSIZE_ANY_EXHDR(t) + VARHDRSZ);
    memcpy(VARDATA(new_t),
           VARDATA_ANY(t),
           VARSIZE_ANY_EXHDR(t));
    PG_RETURN_TEXT_P(new_t);
}
```

Register these functions in SQL:

```sql
CREATE FUNCTION add_one(integer) RETURNS integer
    AS '$libdir/myext', 'add_one'
    LANGUAGE C STRICT;

CREATE FUNCTION copytext(text) RETURNS text
    AS '$libdir/myext', 'copytext'
    LANGUAGE C STRICT;
```

<Warning>
Never modify the contents of a pass-by-reference input value. The pointer you receive may point directly into a disk buffer. Allocate a new copy with `palloc` if you need to modify the value.
</Warning>

## Set-returning C functions (ValuePerCall mode)

C set-returning functions (SRFs) are called repeatedly, returning one row per call. Include `funcapi.h` to use the helper macros.

```c
#include "postgres.h"
#include "fmgr.h"
#include "funcapi.h"

PG_MODULE_MAGIC;

PG_FUNCTION_INFO_V1(generate_series_int);

Datum
generate_series_int(PG_FUNCTION_ARGS)
{
    FuncCallContext *funcctx;
    int32            result;

    if (SRF_IS_FIRSTCALL()) {
        MemoryContext oldcontext;

        /* Initialize on first call */
        funcctx = SRF_FIRSTCALL_INIT();
        oldcontext = MemoryContextSwitchTo(funcctx->multi_call_memory_ctx);

        funcctx->max_calls = (uint64) PG_GETARG_INT32(0);

        MemoryContextSwitchTo(oldcontext);
    }

    /* Every call (including first) */
    funcctx = SRF_PERCALL_SETUP();

    if (funcctx->call_cntr < funcctx->max_calls) {
        result = (int32) (funcctx->call_cntr + 1);
        SRF_RETURN_NEXT(funcctx, Int32GetDatum(result));
    } else {
        SRF_RETURN_DONE(funcctx);
    }
}
```

```sql
CREATE FUNCTION generate_series_int(n integer)
    RETURNS SETOF integer
    AS '$libdir/myext', 'generate_series_int'
    LANGUAGE C STRICT;

SELECT * FROM generate_series_int(5);  -- 1 2 3 4 5
```

### SRF macro reference

| Macro | Purpose |
|-------|---------|
| `SRF_IS_FIRSTCALL()` | True on the first invocation |
| `SRF_FIRSTCALL_INIT()` | Initialize `FuncCallContext`; call only on first invocation |
| `SRF_PERCALL_SETUP()` | Retrieve `FuncCallContext`; call on every invocation |
| `SRF_RETURN_NEXT(funcctx, datum)` | Return one row and request another call |
| `SRF_RETURN_DONE(funcctx)` | Signal end of result set |

The `FuncCallContext` struct fields:

| Field | Description |
|-------|-------------|
| `call_cntr` | Number of calls so far (incremented by `SRF_RETURN_NEXT`) |
| `max_calls` | Optional maximum call count |
| `user_fctx` | Pointer to arbitrary user context data |
| `multi_call_memory_ctx` | Memory context that survives across calls |
| `tuple_desc` | For composite-returning SRFs |
| `attinmeta` | For C-string-based composite construction |

<Note>
SRFs in ValuePerCall mode may not be called to completion (e.g., due to `LIMIT`). Do not rely on the final `SRF_RETURN_DONE` call for cleanup. For functions that hold external resources, use Materialize mode instead.
</Note>

## Variable-length types in C

All variable-length data in PostgreSQL begins with a 4-byte length header. Use these macros:

```c
/* Size of the data excluding the header */
VARSIZE_ANY_EXHDR(ptr)

/* Pointer to the data area */
VARDATA_ANY(ptr)

/* Allocate and fill a varlena value */
text *result = (text *) palloc(VARHDRSZ + nbytes);
SET_VARSIZE(result, VARHDRSZ + nbytes);
memcpy(VARDATA(result), src, nbytes);
```

Never set the length field directly — always use `SET_VARSIZE()`.
