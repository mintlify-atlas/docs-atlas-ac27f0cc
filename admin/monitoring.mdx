title: Monitoring database activity
description: Monitor PostgreSQL using pg_stat_activity, pg_stat_database, pg_stat_user_tables, pg_stat_bgwriter, pg_locks, and pg_stat_replication. Configure track_counts and track_io_timing, use log-based monitoring, and enable auto_explain.
---

PostgreSQL provides rich instrumentation through its **cumulative statistics system** and a set of dynamic views. Most of this data is collected in shared memory and updated periodically by backend processes.

## Enabling statistics collection

Control what statistics are collected in `postgresql.conf`:

```ini postgresql.conf
track_activities = on         # track current query in pg_stat_activity (default on)
track_counts = on             # track row and block access counts (default on; required for autovacuum)
track_io_timing = on          # track I/O timing (small overhead; use on fast storage)
track_wal_io_timing = on      # track WAL I/O timing
track_functions = all         # track function call counts and time (pl, all, or none)
```

<Note>
  `track_io_timing` requires the OS to support high-resolution timing. The overhead is negligible on most modern Linux systems with POSIX timers. Enable it to populate `blk_read_time` and `blk_write_time` in `pg_stat_user_tables` and `pg_stat_database`.
</Note>

## Current activity: pg_stat_activity

`pg_stat_activity` shows one row per server process, including client connections.

```sql
SELECT pid,
       usename,
       application_name,
       client_addr,
       state,
       wait_event_type,
       wait_event,
       query_start,
       now() - query_start AS query_duration,
       left(query, 80) AS query_excerpt
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY query_start;
```

Key `state` values: `active`, `idle`, `idle in transaction`, `idle in transaction (aborted)`, `fastpath function call`, `disabled`.

### Find long-running queries

```sql
SELECT pid, now() - query_start AS duration, query, state
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > interval '5 minutes'
ORDER BY duration DESC;
```

### Find blocking and blocked sessions

```sql
SELECT blocked.pid AS blocked_pid,
       blocked.query AS blocked_query,
       blocker.pid AS blocker_pid,
       blocker.query AS blocker_query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocker
  ON blocker.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;
```

### Terminate a specific session

```sql
-- Gracefully cancel current query
SELECT pg_cancel_backend(pid);

-- Forcibly terminate the connection
SELECT pg_terminate_backend(pid);
```

## Lock monitoring: pg_locks

```sql
SELECT l.pid,
       l.locktype,
       l.mode,
       l.granted,
       c.relname
FROM pg_locks l
LEFT JOIN pg_class c ON c.oid = l.relation
WHERE NOT l.granted
ORDER BY l.pid;
```

To see all lock conflicts:

```sql
SELECT blocked.pid,
       blocked_locks.locktype,
       blocked_locks.mode AS waiting_mode,
       blocking_locks.mode AS blocking_mode,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked
  ON blocked.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.relation = blocked_locks.relation
  AND blocking_locks.pid != blocked_locks.pid
  AND blocking_locks.granted
JOIN pg_catalog.pg_stat_activity blocking_activity
  ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;
```

## Database-level statistics: pg_stat_database

```sql
SELECT datname,
       numbackends,
       xact_commit,
       xact_rollback,
       blks_read,
       blks_hit,
       round(blks_hit::numeric / nullif(blks_hit + blks_read, 0) * 100, 2) AS cache_hit_pct,
       tup_inserted,
       tup_updated,
       tup_deleted,
       conflicts,
       deadlocks,
       blk_read_time,
       blk_write_time
FROM pg_stat_database
WHERE datname = current_database();
```

A `cache_hit_pct` below 99% on a production system often indicates that `shared_buffers` or `effective_cache_size` needs tuning.

## Table statistics: pg_stat_user_tables

```sql
SELECT relname,
       seq_scan,
       seq_tup_read,
       idx_scan,
       idx_tup_fetch,
       n_tup_ins,
       n_tup_upd,
       n_tup_del,
       n_live_tup,
       n_dead_tup,
       round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 2) AS dead_pct,
       last_vacuum,
       last_autovacuum,
       last_analyze,
       last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

Tables with high `dead_pct` and no recent vacuum activity may need manual `VACUUM` or autovacuum tuning.

## I/O statistics: pg_statio_user_tables

```sql
SELECT relname,
       heap_blks_read,
       heap_blks_hit,
       round(heap_blks_hit::numeric / nullif(heap_blks_hit + heap_blks_read, 0) * 100, 2) AS heap_cache_pct,
       idx_blks_read,
       idx_blks_hit
FROM pg_statio_user_tables
ORDER BY heap_blks_read DESC
LIMIT 20;
```

## Index statistics: pg_stat_user_indexes

```sql
SELECT s.relname,
       s.indexrelname,
       s.idx_scan,
       s.idx_tup_read,
       s.idx_tup_fetch,
       pg_size_pretty(pg_relation_size(s.indexrelid)) AS index_size
FROM pg_stat_user_indexes s
ORDER BY s.idx_scan ASC
LIMIT 20;
```

Indexes with zero or very low `idx_scan` counts are unused and may be candidates for removal (verify first — some indexes serve write-side uniqueness constraints even without read scans).

## Background writer: pg_stat_bgwriter

```sql
SELECT checkpoints_timed,
       checkpoints_req,
       checkpoint_write_time,
       checkpoint_sync_time,
       buffers_checkpoint,
       buffers_clean,
       maxwritten_clean,
       buffers_backend,
       buffers_backend_fsync,
       buffers_alloc,
       stats_reset
FROM pg_stat_bgwriter;
```

A high `checkpoints_req` (checkpoint triggered by WAL size, not schedule) suggests `max_wal_size` is too small. High `buffers_backend` (backends writing dirty buffers) suggests `shared_buffers` or bgwriter settings need adjustment.

## Replication monitoring: pg_stat_replication

```sql
SELECT application_name,
       state,
       sent_lsn,
       replay_lsn,
       pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS replay_lag_bytes,
       replay_lag,
       sync_state
FROM pg_stat_replication;
```

See [streaming replication](/admin/streaming-replication) for full details on replication monitoring.

## Log-based monitoring

### Log slow queries

```ini postgresql.conf
log_min_duration_statement = 1000   # log queries over 1 second
log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a '
```

### Log lock waits

```ini
log_lock_waits = on
deadlock_timeout = 1s
```

### Log temporary file creation

```ini
log_temp_files = 0   # log all temp files; set to a size threshold in kB to reduce noise
```

## auto_explain

The `auto_explain` module automatically logs query plans for slow queries without requiring manual `EXPLAIN` instrumentation.

```ini postgresql.conf
shared_preload_libraries = 'auto_explain'
auto_explain.log_min_duration = 1000   # log plans for queries over 1 second
auto_explain.log_analyze = on          # include actual rows and timing
auto_explain.log_buffers = on          # include buffer usage
auto_explain.log_verbose = off         # additional output levels
auto_explain.log_nested_statements = on # log plans for statements inside functions
```

Load it for a single session without a restart:

```sql
LOAD 'auto_explain';
SET auto_explain.log_min_duration = 0;
SET auto_explain.log_analyze = on;
-- Run queries; plans appear in the server log.
```

<Note>
  `auto_explain.log_analyze = on` executes each query and records actual row counts and timing. This has no extra cost since the query runs anyway, but the log volume can be high. Set `log_min_duration` to a meaningful threshold.
</Note>

## Resetting statistics

```sql
-- Reset statistics for the current database
SELECT pg_stat_reset();

-- Reset bgwriter statistics
SELECT pg_stat_reset_shared('bgwriter');

-- Reset statistics for a specific table
SELECT pg_stat_reset_single_table_counters('orders'::regclass);
```
