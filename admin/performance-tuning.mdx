Query performance in PostgreSQL depends on three things: how the query planner chooses to execute the query, whether appropriate indexes exist, and whether server configuration matches the hardware.

## EXPLAIN: reading query plans

The `EXPLAIN` command shows the **query plan** PostgreSQL will use to execute a query.

```sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;
```

Example output:

```
Index Scan using orders_customer_id_idx on orders  (cost=0.42..8.44 rows=1 width=128)
  Index Cond: (customer_id = 42)
```

The numbers in `(cost=startup_cost..total_cost rows=estimated_rows width=avg_row_width_bytes)` are:

- **startup_cost**: estimated cost before the first row is returned (e.g., sorting)
- **total_cost**: estimated cost to return all rows (in arbitrary planner cost units; 1.0 = one sequential page read)
- **rows**: estimated number of rows returned
- **width**: estimated average row size in bytes

## EXPLAIN ANALYZE: actual vs. estimated

`EXPLAIN ANALYZE` actually executes the query and records real row counts and timing.

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 42;
```

Example output:

```
Index Scan using orders_customer_id_idx on orders
  (cost=0.42..8.44 rows=1 width=128)
  (actual time=0.045..0.048 rows=1 loops=1)
  Index Cond: (customer_id = 42)
  Buffers: shared hit=3
Planning Time: 0.234 ms
Execution Time: 0.071 ms
```

When `actual rows` differs significantly from `rows` (estimated), it indicates stale statistics. Run `ANALYZE` to update them.

Common `EXPLAIN` options:

| Option | Description |
|--------|-------------|
| `ANALYZE` | Execute and show actual rows and timing |
| `BUFFERS` | Show shared/local buffer hits and reads (requires `ANALYZE`) |
| `VERBOSE` | Show additional detail including column output list |
| `FORMAT JSON` | Machine-readable JSON output |
| `FORMAT TEXT` | Human-readable text (default) |

<Warning>
  `EXPLAIN ANALYZE` **executes** the query. Wrap destructive queries in a transaction and roll back:

  ```sql
  BEGIN;
  EXPLAIN ANALYZE DELETE FROM orders WHERE status = 'abandoned';
  ROLLBACK;
  ```
</Warning>

## Scan node types

| Node type | When used |
|-----------|-----------|
| **Seq Scan** | Full sequential scan of the table. Used when a large fraction of rows will be returned or no suitable index exists. |
| **Index Scan** | Traverses a B-tree index to locate row positions, then fetches rows from the heap. Efficient for highly selective queries. |
| **Bitmap Index Scan + Bitmap Heap Scan** | Collects matching row locations from the index into a bitmap, then reads heap pages in physical order. Used for moderately selective queries; more efficient than an index scan for returning many rows. |
| **Index Only Scan** | Reads all needed column values directly from the index without touching the heap, if the visibility map shows the data is current. |

## Join node types

| Node type | When used |
|-----------|-----------|
| **Nested Loop** | Iterates over the outer relation and for each row, scans the inner relation. Efficient when the inner relation is small or accessed via index. O(N×M) in the worst case. |
| **Hash Join** | Builds a hash table from the smaller relation, then probes it with each row of the larger relation. Efficient for equi-joins when `work_mem` is sufficient. |
| **Merge Join** | Sorts both relations and merges them. Efficient when both inputs are already sorted or have a useful index. |

## pg_stat_statements: workload analysis

`pg_stat_statements` tracks execution statistics for every distinct query. It is the primary tool for identifying which queries consume the most time or resources.

### Setup

```ini postgresql.conf
shared_preload_libraries = 'pg_stat_statements'
pg_stat_statements.track = all
pg_stat_statements.max = 10000
```

Restart required. Then:

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

### Finding slow queries

```sql
-- Top 20 queries by total execution time
SELECT left(query, 80) AS query_excerpt,
       calls,
       round(total_exec_time::numeric, 2) AS total_ms,
       round(mean_exec_time::numeric, 2) AS mean_ms,
       round(stddev_exec_time::numeric, 2) AS stddev_ms,
       rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

### Finding I/O-heavy queries

```sql
SELECT left(query, 80) AS query_excerpt,
       calls,
       shared_blks_read,
       shared_blks_hit,
       round(shared_blks_hit::numeric /
             nullif(shared_blks_hit + shared_blks_read, 0) * 100, 2) AS cache_hit_pct
FROM pg_stat_statements
ORDER BY shared_blks_read DESC
LIMIT 20;
```

### Reset statistics

```sql
SELECT pg_stat_statements_reset();
```

## Memory configuration

### shared_buffers: 25% of RAM

`shared_buffers` is the shared in-memory cache for data pages. Setting it too low causes excess disk I/O; setting it too high leaves less memory for OS file cache.

```ini
shared_buffers = 2GB    # for an 8 GB system
```

### work_mem: per sort/hash operation

`work_mem` controls the memory available for each sort or hash operation within a query. A complex query may allocate multiple `work_mem` chunks simultaneously, and `max_connections` connections can each run multiple operations.

```ini
work_mem = 32MB    # reasonable starting point
```

Increase `work_mem` if you see temporary files created for sort operations:

```sql
SELECT query, temp_blks_written
FROM pg_stat_statements
ORDER BY temp_blks_written DESC
LIMIT 10;
```

### effective_cache_size: 75% of RAM

`effective_cache_size` is a hint to the planner about the total amount of memory available for disk caching (OS page cache plus `shared_buffers`). It does not allocate memory.

```ini
effective_cache_size = 6GB    # for an 8 GB system
```

A correct `effective_cache_size` helps the planner prefer index scans over sequential scans for larger datasets.

### maintenance_work_mem

Used by `VACUUM`, `CREATE INDEX`, `ALTER TABLE ADD FOREIGN KEY`. Can be set higher than `work_mem` since only a few maintenance operations run at a time.

```ini
maintenance_work_mem = 512MB
```

## Planner cost parameters for SSDs

```ini postgresql.conf
random_page_cost = 1.1           # close to seq_page_cost for SSDs
effective_io_concurrency = 200   # number of concurrent I/Os SSDs can handle
```

With spinning disks, keep the defaults: `random_page_cost = 4.0` and `effective_io_concurrency = 1` (or the number of spindles).

## Indexes

### Partial indexes

A partial index covers only rows matching a `WHERE` condition, making it smaller and faster:

```sql
-- Index only active orders (much smaller than indexing all orders)
CREATE INDEX CONCURRENTLY orders_active_idx
  ON orders (created_at)
  WHERE status = 'active';
```

The planner will use this index only for queries that include the condition in the `WHERE` clause.

### Expression indexes

An expression index indexes the result of an expression rather than a raw column value:

```sql
-- Case-insensitive search
CREATE INDEX CONCURRENTLY users_email_lower_idx
  ON users (lower(email));

-- Query that uses the index:
SELECT * FROM users WHERE lower(email) = lower('Alice@Example.COM');
```

### Covering indexes (INCLUDE)

Add non-key columns to an index to enable index-only scans:

```sql
CREATE INDEX CONCURRENTLY orders_customer_covering_idx
  ON orders (customer_id)
  INCLUDE (total_amount, status, created_at);
```

Queries that need only `customer_id`, `total_amount`, `status`, and `created_at` can be satisfied entirely from the index without touching the heap.

### Multi-column indexes

```sql
-- Most selective column should be first
CREATE INDEX orders_compound_idx ON orders (status, created_at);
```

## Connection pooling

Each PostgreSQL backend process consumes memory (typically 5–10 MB). With many clients, the overhead of per-process memory and context switching reduces throughput.

**Recommendation**: Use PgBouncer in front of PostgreSQL for applications with many short-lived connections. Set `pool_mode = transaction` for the most efficient pooling, but verify application compatibility.

```ini pgbouncer.ini
max_client_conn = 1000
default_pool_size = 25   # actual backend connections per database/user pair
pool_mode = transaction
```

This allows 1,000 client connections while maintaining only 25 actual backend connections to PostgreSQL.

## Query tuning checklist

<Steps>
  <Step title="Run EXPLAIN ANALYZE">
    Identify the slowest part of the query plan. Look for nodes with large discrepancies between estimated and actual rows.
  </Step>

  <Step title="Check for missing indexes">
    Sequential scans on large tables with selective `WHERE` conditions indicate a missing index.

    ```sql
    EXPLAIN SELECT * FROM orders WHERE customer_id = 42;
    -- Seq Scan = no usable index; create one:
    CREATE INDEX CONCURRENTLY orders_customer_idx ON orders (customer_id);
    ```
  </Step>

  <Step title="Update statistics if estimates are wrong">
    ```sql
    ANALYZE orders;
    -- Rerun EXPLAIN ANALYZE and check if estimates improved.
    ```
  </Step>

  <Step title="Increase work_mem if sorts spill to disk">
    ```sql
    SET work_mem = '128MB';
    EXPLAIN ANALYZE SELECT ...
    -- Recheck if Sort node still shows 'Sort Method: external merge  Disk'
    ```
  </Step>

  <Step title="Tune planner cost parameters for your storage">
    Set `random_page_cost = 1.1` and `effective_io_concurrency = 200` if running on SSDs.
  </Step>
</Steps>
