Data Definition Language (DDL) is the subset of SQL used to define and modify database structure. This page covers the most important DDL operations in PostgreSQL: creating and modifying tables, organizing objects with schemas, and an introduction to inheritance and partitioning.

## CREATE TABLE

The `CREATE TABLE` command defines a new table with its columns and data types.

```sql
CREATE TABLE products (
    product_no  integer,
    name        text,
    price       numeric DEFAULT 9.99
);
```

### Default values

A column default is evaluated at insert time, not when the table is created.

```sql
CREATE TABLE events (
    event_id  bigint GENERATED ALWAYS AS IDENTITY,
    created_at timestamptz DEFAULT CURRENT_TIMESTAMP,
    payload   jsonb
);
```

### Generated columns

Generated columns are computed from other columns. A `STORED` generated column is written to disk; a `VIRTUAL` generated column (the default) is computed on read.

```sql
CREATE TABLE people (
    height_cm numeric,
    height_in numeric GENERATED ALWAYS AS (height_cm / 2.54) STORED
);
```

<Note>
A generated column cannot be written to directly, cannot use volatile functions, and cannot reference other generated columns.
</Note>

### Identity columns

Identity columns use an implicit sequence. `GENERATED ALWAYS` rejects explicit values unless `OVERRIDING SYSTEM VALUE` is specified; `GENERATED BY DEFAULT` allows them.

```sql
CREATE TABLE orders (
    order_id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id integer NOT NULL,
    total numeric
);
```

## ALTER TABLE

`ALTER TABLE` modifies an existing table's definition.

<AccordionGroup>
  <Accordion title="Add, drop, and rename columns">

```sql
-- Add a column with a default
ALTER TABLE products ADD COLUMN weight numeric DEFAULT 0;

-- Drop a column
ALTER TABLE products DROP COLUMN weight;

-- Rename a column
ALTER TABLE products RENAME COLUMN name TO product_name;
```

  </Accordion>

  <Accordion title="Change column type">

```sql
ALTER TABLE products
    ALTER COLUMN price TYPE double precision;
```

PostgreSQL will attempt an implicit cast. Provide an explicit `USING` expression when a cast is not available.

```sql
ALTER TABLE logs
    ALTER COLUMN ip_address TYPE inet
    USING ip_address::inet;
```

  </Accordion>

  <Accordion title="Add and drop constraints">

```sql
ALTER TABLE products ADD CONSTRAINT positive_price CHECK (price > 0);
ALTER TABLE products DROP CONSTRAINT positive_price;

ALTER TABLE orders ADD PRIMARY KEY (order_id);
ALTER TABLE order_items
    ADD FOREIGN KEY (product_no) REFERENCES products;
```

  </Accordion>

  <Accordion title="Rename and set schema">

```sql
ALTER TABLE products RENAME TO inventory;
ALTER TABLE inventory SET SCHEMA warehouse;
```

  </Accordion>
</AccordionGroup>

## DROP TABLE

```sql
DROP TABLE products;

-- Silently skip if the table does not exist
DROP TABLE IF EXISTS products;

-- Also drop dependent objects (views, foreign keys referencing this table)
DROP TABLE products CASCADE;
```

<Warning>
`DROP TABLE CASCADE` removes all objects that depend on the table, including foreign key constraints in other tables. Use with care in production.
</Warning>

## Schemas

Schemas are namespaces within a database. Every table belongs to a schema; the default is `public`.

```sql
-- Create a schema
CREATE SCHEMA sales;

-- Create a table inside a schema
CREATE TABLE sales.orders (
    order_id integer PRIMARY KEY,
    placed_at timestamptz DEFAULT CURRENT_TIMESTAMP
);

-- Set the schema search path for the session
SET search_path TO sales, public;

-- Grant usage so other roles can see the schema
GRANT USAGE ON SCHEMA sales TO reporting_role;
```

### System catalog schema

`pg_catalog` is always searched before any user schema. It contains all built-in types, functions, and operators.

<Tip>
Avoid naming your tables or columns with the `pg_` prefix to prevent future conflicts with system catalog additions.
</Tip>

## Tablespaces

Tablespaces map a logical name to a directory on the file system, allowing data to be placed on different storage media.

```sql
-- Create a tablespace (requires superuser; path must exist and be owned by the postgres OS user)
CREATE TABLESPACE fast_storage
    OWNER postgres
    LOCATION '/mnt/ssd/pgdata';

-- Create a table on that tablespace
CREATE TABLE high_frequency_events (
    id bigint GENERATED ALWAYS AS IDENTITY,
    ts timestamptz,
    data jsonb
) TABLESPACE fast_storage;

-- Move an existing table
ALTER TABLE high_frequency_events SET TABLESPACE pg_default;
```

## Inheritance

PostgreSQL supports table inheritance: a child table inherits all columns of its parent.

```sql
CREATE TABLE cities (
    name       text,
    population float,
    elevation  int
);

CREATE TABLE capitals (
    state char(2)
) INHERITS (cities);
```

A query against `cities` returns rows from both `cities` and `capitals`. Use `ONLY` to query only the parent:

```sql
-- Returns cities AND capitals
SELECT name, elevation FROM cities WHERE elevation > 500;

-- Returns only non-capital cities
SELECT name, elevation FROM ONLY cities WHERE elevation > 500;
```

<Note>
Check constraints and NOT NULL constraints are inherited automatically. UNIQUE, PRIMARY KEY, and FOREIGN KEY constraints are not.
</Note>

## Partitioning overview

Declarative partitioning splits a large table into smaller physical partitions while presenting a single logical table to queries.

```sql
-- Declare a range-partitioned table
CREATE TABLE measurement (
    city_id   int  NOT NULL,
    logdate   date NOT NULL,
    peaktemp  int,
    unitsales int
) PARTITION BY RANGE (logdate);

-- Create individual partitions
CREATE TABLE measurement_2024_01 PARTITION OF measurement
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE measurement_2024_02 PARTITION OF measurement
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Partition-level index (applied to all current and future partitions)
CREATE INDEX ON measurement (logdate);
```

PostgreSQL supports three partitioning methods:

<CardGroup cols={3}>
  <Card title="RANGE" icon="arrows-left-right">
    Rows are assigned to the partition whose range includes the partition key value. Useful for time-series and sequential IDs.
  </Card>
  <Card title="LIST" icon="list">
    Each partition holds an explicit list of key values. Useful for region codes, status values, or enumerated categories.
  </Card>
  <Card title="HASH" icon="hashtag">
    Rows are distributed across partitions based on a hash of the key modulo the partition count. Useful for even data distribution.
  </Card>
</CardGroup>

For a complete partitioning reference including DEFAULT partitions, partition pruning, and partition-wise joins, see [Partitioning](/sql/partitioning).
